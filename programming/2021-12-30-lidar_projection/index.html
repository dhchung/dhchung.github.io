LiDAR Projection

30 Dec 2021 in programming

Projecting lidar onto image plane

LiDAR - camera calibration program을 제작하던 중, projection에 해당되는 픽셀값을 가져와 LiDAR color에 적용시키는데 시간이 많이 드는 것을 확인했다.

Ouster OS1 64채널 기준 point 수: 65536 Ouster OS2 128채널 기준 point 수: 262144

일단 우리가 기본적으로 배우는 것과 같이 LiDAR의 Extrinsic parameter를 통한 transformation matrix와 Camaera의 projection matrix를 다음과 같이 정의하면:

LiDAR Extrinsic parameter: \(R_{LiDAR}, t_{LiDAR}\)

LiDAR Transformation matrix: \(T_{LiDAR} = \left[ R\; \vert \; t; \left[ 0, 0, 0, 1 \right] \right]\)

Camera Extrinsic parameter: \(R_{cam}, t_{cam}\)

Camera Projection matrix: \(P = C \left [ R' \; \vert \; -R't \right]\)

위와 같이 되고, LiDAR point cloud를 \(\mathbf{X}_{LiDAR}\)로 표현하게 되면:

LiDAR point cloud : \(\mathbf{X}_{LiDAR} = \left[ \mathbf{x}_0, \mathbf{x}_1, \dots , \mathbf{x}_n \right]\)

LiDAR point cloud in global reference frame: \(\mathbf{X}h_{Global} = T_{LiDAR}P_{LiDAR}\)

여기서 \(\mathbf{X}h\)는 homogeneous coordinate인데, 실제로 4번째 row만 없애면 된다.

어쨌든, 카메라에 projection된 LiDAR point cloud는 다음과 같이 된다.

Projected Lidar point cloud : \(\mathbf{X}h_{cam} = P\mathbf{X}h_{Global}\)

이건 다시 풀어쓰면

Projected Lidar point cloud:

\[\mathbf{X}h_{cam} = C \left[ R' \; \vert \; -R't \right] T_{LiDAR}P_{LiDAR}\]

만약 LiDAR point cloud를 먼저 global reference frame로 바꿔주고 projection matrix를 다시 곱해준다면 연산이 2배 이상으로 드는 것을 확인할 수 있다.

그래서 일단 Projection matrix와 lidar의 transformation matrix를 먼저 곱해주고 pointcloud에 적용하면 더 효율적인 projection이 될 것이다.

테스트는 Eigen을 사용했다.

Transformation matrix: Eigen::Matrix4f

Lidar Point Cloud: Eigen::Matrix4Xf

OS2 128ch의 point cloud transformation 시간: 0.0647862s

OS1 64ch의 point cloud transformation 시간 : 0.0309353s

OS2 128ch의 point projection 시간 : 0.502041s

OS1 64ch의 point projection 시간: 0.253995s

무려 9~10배가량 시간이 뛴것을 확인할 수 있었다! 여기서 놀라운건 point transformation -> projection = point projection이 아니라 그냥 projection에만 시간이 이렇게 뛴 것을 확인할 수 있다.

Projection Matrix에는 3 by 4의 Eigen::MatrixXf를 사용하였는데…. 어떡하지 ㅜ Eigen::Matrix4f를 사용해볼까 한다.

Projection Matrix를 4 by 4의 Eigen::Matrix4f를 사용해서 projection 결과를 기존 \(\left[u; v; w\right]\)에서 \(\left[u; v; w; 1\right]\)로 쓸데없지만 마지막에 1을 넣게 되었다.

그러니까 다음과 같았다.

OS2 128ch의 point projection 시간 : 0.0243272s

OS1 64ch의 point projection 시간: 0.0122136s

솔까 이해안됨 근데 이종의 matrix type에 대해서 계산시간이 확실히 차이가 나는 것 같다.

OS2 128ch의 point cloud transformation 시간: 0.0647862s(debug), 0.00182863(RelWithDebInfo)

OS1 64ch의 point cloud transformation 시간 : 0.0309353s(debug), 0.00104007s(RelWithDebInfo)

OS2 128ch의 point projection 시간 : 0.502041s (debug), 0.000720223s (RelWithDebInfo)

OS1 64ch의 point projection 시간: 0.253995s (debug), 0.000356825s (RelWithDebInfo)

Point Projection에 Eigen::Matrix4f를 사용하면

OS2 128ch의 point projection 시간 : 0.0243272s (debug), 0.000662277s (RelWithDebInfo)

OS1 64ch의 point projection 시간: 0.0122136s (debug), 0.000243122s (RelWithDebInfo)

확실히 debug모드에선 이종 matrix에 대해 시간이 더 걸리나보다. 여튼 RelWithDebInfo에서는 큰 차이가 없으나, 이종에 대해서는 시간이 더 걸리는 것을 확인할 수 있다.

여기서도 신기한건, transformation이랑 projection이랑 거의 시간이 비슷할 것 같은데 차이가 3~5배 정도 차이가 난다…………………. 모르겄다 이젠

여튼 아주 작은 시간이라도 (ms) 아까운 법, transformation 후 projection 시키면 다음과 같다:

OS2 128ch Total time: 0.00243003s

OS1 64ch Total time: 0.00169765s

Projection matrix와 transformation matrix를 먼저 곱한 뒤 projection 시키면 다음과 같다:

OS2 128ch Total time: 0.000769061s OS1 64ch Total time: 0.000388105s

거의 무슨 3배~5배정도 더 빨라진다.

여튼 결론:

point cloud에 대해서 두번 곱하기 때문에 시간이 오래 걸리는 것인 줄 알았던 문제가 compile 모드에 의한 것이었다는 것을 깨달음

Eigen::MatrixXf나 Eigen::Matrix3Xf 를 사용하는 것보다 크기가 정해진 Eigen::Matrix4f를 사용하는 것이 더 빠름
